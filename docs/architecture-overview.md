# Архитектура проекта

## Зачем этот документ

Этот файл нужен как короткая карта проекта для агента или разработчика, который впервые заходит в репозиторий. Он не заменяет README и не повторяет все детали модулей, а даёт общий mental model: из каких частей состоит система, где находятся источники истины и как проходят основные потоки данных.

## Система целиком

Проект состоит из трёх основных частей:

1. `apps/web/` — статический SPA на ванильном JavaScript и нативных ES-модулях.
2. `apps/api/` — Node.js API на Express.
3. SQLite база данных, путь к которой задаётся через `DB_PATH`.

Высокоуровневая схема:

```text
Браузер
  -> apps/web (static files)
  -> HTTP calls to /tasks/api or /tasks-stg/api
        -> Nginx
        -> Node.js API (Express)
        -> SQLite
```

На production и staging архитектура одинаковая, различаются только URL, порты, unit-файлы и путь деплоя.

## Главные точки входа

### Frontend

- `apps/web/index.html` — HTML-оболочка.
- `apps/web/main.js` — orchestration слой фронтенда: импортирует модули, хранит часть runtime state и регистрирует callback-зависимости между модулями.
- `apps/web/style.css` — глобальные стили.

### Backend

- `apps/api/src/server.js` — запуск HTTP-сервера и фонового lifecycle для рабочего дня.
- `apps/api/src/app.js` — сборка Express-приложения, middleware и маршрутов.
- `apps/api/src/config/index.js` — чтение окружения, включая `DB_PATH`.
- `apps/api/src/db/client.js` — тонкая обёртка над CLI `sqlite3`.

## Репозиторий по зонам ответственности

- `apps/web/` — клиентская логика, UI, локальное хранилище, синхронизация с API.
- `apps/api/` — HTTP API, доменная логика, SQL-операции, миграции, тесты.
- `infra/nginx/` — эталонный nginx-конфиг для `/tasks` и `/tasks-stg`.
- `infra/systemd/` — эталонные unit-файлы для `tasks-prod` и `tasks-stg`.
- `.github/workflows/` — деплой на staging и promote в production.
- `FRONTEND-MODULARIZATION.md` — исторический документ о разбиении фронтенда на модули.

## Источники истины

### В режиме `local`

- Задачи, проекты, архив, рабочий день и часть UI-настроек живут в `localStorage`.
- API не нужен.
- Фронтенд работает как полностью локальное приложение.

### В режиме `server`

- Источником истины для доменных данных становится API + SQLite.
- `localStorage` остаётся вспомогательным слоем для:
  - выбора режима хранения,
  - API-ключа,
  - кэша рабочего дня,
  - кэша активных таймеров,
  - UI-настроек.
- Фронтенд отправляет CRUD-операции в API и при необходимости перезагружает данные с сервера.

Важно: проект поддерживает оба режима одновременно в одном UI, поэтому многие модули фронтенда написаны с развилкой `local` vs `server`.

## Как frontend выбирает окружение

`apps/web/src/config.js` вычисляет API-префикс по `location.pathname`:

- путь начинается с `/tasks-stg` -> API `/tasks-stg/api`
- иначе -> API `/tasks/api`

Это означает:

- один и тот же фронтенд-код обслуживает STG и PROD;
- окружение определяется URL страницы, а не отдельной сборкой.

## Как устроен backend

Express-приложение собирается в `apps/api/src/app.js`:

1. `express.json`
2. request logger
3. CORS middleware
4. public route: `/api/health`
5. auth middleware на весь остальной `/api`
6. доменные роуты:
   - `/api/projects`
   - `/api/tasks`
   - `/api/archive`
   - `/api/workday`
   - `/api/import`
   - `/api/stats`
   - `/api/yearplan`
7. глобальный error handler

Аутентификация простая: статический API-ключ в заголовке `X-API-Key`.

## Как устроена база

API не использует npm-драйвер для SQLite. Вместо этого `apps/api/src/db/client.js` вызывает системный бинарь `sqlite3` через `execFile`.

С практической точки зрения это значит:

- SQL остаётся очень явным и близким к миграциям;
- транзакции собираются вручную через батч SQL-выражений;
- при разворачивании окружения важна доступность бинаря `sqlite3`.

Основные таблицы:

- `projects`
- `tasks`
- `archive`
- `workdays`
- `yearplan_activities`
- `meta`

Подробности по полям и инвариантам описаны в [data-model.md](./data-model.md).

## Ключевые домены

### Tasks

Главная сущность приложения. Поддерживается иерархия до двух уровней вложенности (`MAX_TASK_DEPTH = 2`, то есть глубина дерева ограничена).

### Projects

Категоризация задач и связка с `yearplan`.

### Workday

Отдельная доменная сущность, а не только UI-фича. Сервер хранит её в таблице `workdays`, умеет открывать, синхронизировать, закрывать и переоткрывать рабочий день.

### Archive

Исторические снапшоты завершённых рабочих дней и/или удалённых поддеревьев задач.

### Year Plan

Годовой календарный слой с активностями, диапазонами дат, цветом, статусом выполнения и опциональной привязкой к проекту.

## Основные runtime-потоки

### Старт фронтенда

1. Браузер загружает `index.html`.
2. `main.js` импортирует модули и регистрирует callback-реестры.
3. Из `localStorage` читаются режим, UI-настройки и локальные данные.
4. Если активен `server` режим, `main.js` запрашивает задачи, проекты и текущий рабочий день через API.

### Старт API

1. Загружается конфиг и `.env`.
2. Поднимается Express-сервер.
3. При первом доступе к БД `db/client.js` создаёт каталог БД при необходимости и включает `WAL`.
4. `workdayLifecycle` запускает фоновый цикл, который периодически проверяет текущее состояние рабочего дня.

### Изменение задачи

1. UI меняет клиентский runtime state.
2. `storage.js` решает, писать ли в `localStorage` или делегировать серверную запись.
3. В `server` режиме `api.js` ставит запрос в очередь или отправляет его сразу.
4. При сбое серверной записи фронтенд может сделать refresh с сервера.

## Почему на фронтенде есть callback-реестры

Во фронтенде избегают жёстких циклических импортов. Поэтому часть модулей экспортирует `register*Callbacks(...)`, а `main.js` связывает их после инициализации.

Это ключевая архитектурная особенность проекта:

- `main.js` — не просто entry point, а композиционный слой;
- модули остаются более изолированными;
- цена за это — необходимость понимать, какие зависимости приходят через регистрацию, а не через `import`.

## Деплой и окружения

### Staging

- workflow: `.github/workflows/deploy-stg.yml`
- URL: `https://parkhomenko.space/tasks-stg/`
- API проксируется на `127.0.0.1:3102`

### Production

- workflow: `.github/workflows/promote-prod.yml`
- URL: `https://parkhomenko.space/tasks/`
- API проксируется на `127.0.0.1:3101`

Код API и фронт доставляются отдельно: API в `/srv/tasks-*`, статика в `/var/www/tasks-*`.
