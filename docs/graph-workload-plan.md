# План реализации вида «График» (месячная нагрузка)

## Цель
Сделать в приложении вид «График» (месячный обзор по будням), где каждый день показывает:
- суммарно затраченное время за день (все задачи, не только завершенные),
- число завершенных задач за день,
- визуальную «эффективность дня» через заливку карточки относительно дневной цели.

Цель рабочего дня:
- единая глобальная для всех рабочих дней,
- хранится на сервере (в API-режиме),
- задается в новом разделе настроек: `Рабочий день`.

## Зафиксированные продуктовые решения
- В графике отображаем только `Пн–Пт`.
- Эффективность считаем по всему времени за день, даже если задача не завершена.
- Для календарной сетки показываем дни соседних месяцев как неактивные.
- Цветовую шкалу и пороги внедряем как часть UX-дизайна фичи.
- Серверное хранение дневной цели: используем таблицу настроек приложения (`app_settings`), без альтернативной схемы в рамках этой фичи.

## Общий подход
Реализация по фазам с ранней ценностью:
1. UI/UX каркаса и визуальной модели прогресса.
2. Проверка и фиксация модели данных по дням (до агрегаций).
3. Источник данных и агрегации на фронте.
4. Настройки цели рабочего дня.
5. Серверное хранение цели и синхронизация с режимами хранения.
6. Доработка качества: тесты, edge-cases, polish.

## Фаза 1. Закрепить вид «График» как стабильный UI-контейнер
### Что делаем
- Доводим структуру карточки дня до финального формата:
  - дата,
  - время,
  - завершенные задачи,
  - слой прогресса (заливка).
- Фиксируем календарную матрицу по рабочим неделям (`Пн–Пт`) для любого месяца.
- Добавляем состояния карточек:
  - `текущий месяц`,
  - `вне месяца` (неактивные),
  - `сегодня` (опционально акцент).

### Результат фазы
- Визуально стабильный календарный вид без перекосов сетки.
- UI готов к подаче реальных данных без изменения структуры.

## Фаза 2. Модель эффективности и визуальная шкала
### Что делаем
- Реализуем на моковых данных (статические `spentMinutes`/`doneCount`) до подключения живой агрегации.
- Вводим формулу:
  - `progress = min(spentMinutes / targetMinutes, 1)`.
- Проектируем цветовые пороги (предложение):
  - `0–39%` — низкая загрузка (нейтрально-слабый тон),
  - `40–79%` — рабочая зона (умеренный тон),
  - `80–99%` — близко к цели (насыщенный тон),
  - `>=100%` — цель выполнена (зеленый, 100% заливки).
- Реализуем заливку карточки через CSS-переменные (процент + тон).
- Для неактивных дней оставляем приглушенный стиль без яркого прогресса.

### Результат фазы
- Карточка визуально считывается за 1–2 секунды.
- Цвет и высота/плотность заливки отражают достижение цели.
- Подключение к реальным данным не является целью этой фазы.

## Фаза 3. Проверка и фиксация модели данных по дням
### Что делаем
- Явно документируем текущее состояние модели задач:
  - есть ли у задачи дата завершения,
  - как хранится время: только суммарно (`timeSpent`) или посуточно.
- Принимаем решение по минимально достаточной модели для `Графика`:
  - если данных достаточно, фиксируем контракт агрегации,
  - если данных недостаточно, проектируем расширение модели (дата закрытия и/или daily stats/worklog).
- Обновляем `docs/data-model.md` и `docs/api-contract.md` под принятое решение.

### Результат фазы
- Снят архитектурный развилок по «времени за день».
- Следующие фазы не содержат скрытой переделки модели.

### Статус на 2026-04-30
- Часть фазы выполнена: фактическое состояние модели зафиксировано в `docs/data-model.md` и `docs/api-contract.md`.
- Подтверждено:
  - `timeSpent` хранится итогом на задаче (без встроенного дневного журнала),
  - `completedAt` есть на API/в БД,
  - `completedAt` теперь протащен в фронтенд-модель задач (нормализация/миграция/toggle done).
- Остаток для следующих фаз:
  - решить и внедрить модель для точного `spent/day` (daily stats/worklog или согласованная эвристика).

## Фаза 4. Агрегация данных на фронте (local + API единым кодом)
### Что делаем
- Выделяем единый селектор/агрегатор:
  - вход: список задач + месяц,
  - выход: map `YYYY-MM-DD -> { spentMinutes, doneCount }`.
- Источник времени:
  - используем модель, утвержденную в Фазе 3,
  - для текущего дня учитываем и активный таймер.
- Источник `doneCount`:
  - считаем задачи, завершенные в конкретный день (если есть поле даты закрытия),
  - если даты закрытия нет: задача переносится в блок расширения модели данных Фазы 3 (не в UI-фазы).

### Результат фазы
- Вид «График» заполняется живыми данными в текущем режиме (`LS`/`API`).
- Логика рендера не знает источник хранения, получает только агрегат.

### Статус на 2026-05-01
- Фаза закрыта:
  - `doneCount/day` считается из `completedAt` с fallback на `updatedAt/createdAt` для старых записей;
  - `spent/day` считается из реального `timeSpent` и активного таймера с anchor-date стратегией (`completedAt -> updatedAt -> createdAt`, для активного таймера — текущий день);
  - одна и та же агрегация работает для `LS` и `API`, потому что использует общее runtime-состояние `tasks`.
- Ограничение (осознанно вынесено в следующие фазы): точный исторический `spent/day` без daily worklog недостижим по текущей модели.

### Результат фазы

### Текущее состояние

- Значит фаза остается открытой.
